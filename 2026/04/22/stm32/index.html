新建工程

首先新建工程后，这个工程不能直接用，需要添加一些必要文件。

添加 start 启动相关文件

打开
\STM32F10x_StdPeriph_Lib_V3.5.0.zip\STM32F10x_StdPeriph_Lib_V3.5.0\Libraries\CMSIS\CM3\DeviceSupport\ST\STM32F10x\startup\arm
这些是 STM32 的启动文件，STM32 由启动文件开始执行。在项目的根目录新建 start 文件夹，把启动文件复制进去。

启动文件一般是类似下面这种名字：

文件名 文件类型 主要作用 通俗理解
startup_stm32f10x_xx.s 汇编文件 单片机上电后最先执行的文件，负责初始化堆栈、设置中断向量表，并最终跳转到 main() 相当于单片机“开机后执行的第一段程序”

启动文件不是随便选的，要和芯片型号对应。STM32F10x 按容量一般分为几类，不同芯片要选不同的启动文件：

启动文件名 适用范围
startup_stm32f10x_ld.s 小容量器件
startup_stm32f10x_md.s 中容量器件
startup_stm32f10x_hd.s 大容量器件
startup_stm32f10x_xl.s 超大容量器件

例如：

芯片型号 常见选择
STM32F103C8T6 通常选 startup_stm32f10x_md.s
STM32F103RCT6 通常选 startup_stm32f10x_hd.s

也就是说，启动文件要和芯片容量匹配，否则中断向量表和存储器配置可能不合适。

接下来打开
\STM32F10x_StdPeriph_Lib_V3.5.0\Libraries\CMSIS\CM3\DeviceSupport\ST\STM32F10x
把 .c 和 .h 三个文件复制到 start 里。

文件名 文件类型 主要作用 通俗理解
stm32f10x.h 头文件 定义 STM32F10x 系列单片机的寄存器、外设基地址、中断号、数据类型等内容，供程序直接操作硬件使用 相当于单片机的“硬件说明书接口”，让代码认识 GPIOA、RCC 这些硬件对象
system_stm32f10x.c 源文件 完成系统底层初始化，主要包括系统时钟配置、PLL 设置、总线时钟分频、向量表设置等 相当于单片机开机后的“系统启动配置程序”，决定芯片按什么主频运行
system_stm32f10x.h 头文件 对 system_stm32f10x.c 中的全局变量和函数进行声明，例如 SystemInit()、SystemCoreClockUpdate()、SystemCoreClock 相当于系统初始化模块的“对外说明书”，让别的文件能调用这些初始化函数

三个文件之间的关系：

文件 关系说明
stm32f10x.h 提供芯片底层硬件定义
system_stm32f10x.c 利用这些底层定义去完成系统初始化
system_stm32f10x.h 把 system_stm32f10x.c 里的函数和变量声明出来，方便其他文件调用

STM32 是由内核和内核外围设备组成，内核寄存器描述文件和外围设备描述文件不在一起，因此还要添加内核寄存器的描述文件。打开
\STM32F10x_StdPeriph_Lib_V3.5.0\Libraries\CMSIS\CM3\CoreSupport
复制两个文件，粘贴到 start 中。

这两个文件通常是：

文件名 文件类型 主要作用 通俗理解
core_cm3.h 头文件 定义 Cortex-M3 内核相关寄存器、NVIC、中断控制、SysTick 等内容 相当于 ARM 内核的“使用说明书”，让程序能操作内核功能
core_cm3.c 源文件 实现部分 Cortex-M3 内核支持函数 相当于对 core_cm3.h 里部分功能的补充实现

这几个 start 文件合起来的作用可以概括为：

文件 作用定位
startup_stm32f10x_xx.s 负责“开机启动”
stm32f10x.h 负责“芯片硬件定义”
system_stm32f10x.c/.h 负责“系统时钟与底层初始化”
core_cm3.h/.c 负责“Cortex-M3 内核支持”

打开 Keil5 上方魔术棒，进入 C/C++，先在 Include Paths 里把 start 文件夹加进去。

创建用户文件夹和主函数文件

创建 user 文件夹和 main.c 文件。

这里可以顺手理解一下工程目录的基本分工：

文件夹/文件 作用 通俗理解
start 启动文件、系统文件、内核文件 让 STM32 能正常上电并进入 main()
user 用户自己编写的应用代码 写实验功能和主逻辑
main.c 主函数文件 程序入口，主要功能从这里开始写

首次编译报错

注意如果这里编译报错，一般是 Keil 版本与旧版 CMSIS 文件不兼容。可以点击 Keil 魔术棒按钮，进入 Code Generation，把编译器改成 Use default compiler version 5。

这类问题尤其容易出现在 core_cm3.c 上。原因是老版本的 CMSIS 文件更适配 ARM Compiler 5，而新版 armclang 对一些旧写法兼容性较差。因此有一个很实用的经验：

文件 处理建议
core_cm3.h 一般要保留
core_cm3.c 某些工程里需要，某些工程里反而会导致兼容报错，要结合编译环境判断

也就是说：

  • core_cm3.h 往往是必须的
  • core_cm3.c 不一定在所有工程里都必须保留
  • 如果出现老 CMSIS 与新编译器不兼容，优先切换到 Compiler Version 5

添加标准外设库文件

仅有 start 和 main.c 还不够，因为实际开发时还要操作 GPIO、串口、定时器、中断等外设，所以还需要把 STM32 标准外设库加入工程。

打开路径：

\STM32F10x_StdPeriph_Lib_V3.5.0\STM32F10x_StdPeriph_Lib_V3.5.0\Libraries\STM32F10x_StdPeriph_Driver

这个目录下通常有两个重要文件夹：

文件夹 作用 通俗理解
inc 外设库头文件 相当于各个外设的“说明书”
src 外设库源文件 相当于各个外设的“具体功能实现”

在项目根目录新建 library 文件夹。把 inc 里的 .h 文件和 src 里需要用到的 .c 文件复制到 library 中。

这些文件的命名规律很统一：

命名形式 含义
stm32f10x_xxx.h xxx 外设的头文件
stm32f10x_xxx.c xxx 外设的驱动实现文件

常见外设文件说明如下：

文件名 作用 通俗理解
stm32f10x_rcc.c/.h 时钟控制 给外设开时钟，不开时钟很多外设无法使用
stm32f10x_gpio.c/.h GPIO 控制 控制引脚输入输出，例如 LED、按键、蜂鸣器
stm32f10x_usart.c/.h 串口通信 用于串口收发数据
stm32f10x_tim.c/.h 定时器 用于定时、计数、PWM 等
stm32f10x_exti.c/.h 外部中断 用于按键中断、外部信号触发中断
stm32f10x_adc.c/.h ADC 采集模拟电压
stm32f10x_spi.c/.h SPI 与 SPI 外设通信
stm32f10x_i2c.c/.h I2C 与 I2C 外设通信
stm32f10x_dma.c/.h DMA 自动搬运数据，减轻 CPU 负担
misc.c/.h 杂项支持 主要用于 NVIC 中断优先级配置

并不是所有外设库文件都要一次性全部加入工程。通常只需要加入当前实验会用到的那些文件即可。加得太多虽然一般也能编译，但工程会更乱，排错也更麻烦。

例如，做最基础的 LED、按键、蜂鸣器实验时，通常只需要下面这些：

常用文件 用途
stm32f10x_rcc.c/.h 开时钟
stm32f10x_gpio.c/.h 控制 GPIO
stm32f10x_exti.c/.h 外部中断
misc.c/.h 配置中断优先级
stm32f10x_tim.c/.h 定时器相关功能

添加用户模板文件

除了 main.c，通常还需要添加用户配置文件和中断文件。可以参考标准库自带模板工程，常见需要添加的文件有：

文件名 作用 通俗理解
main.c 主函数文件 程序入口，主要功能从这里开始写
stm32f10x_conf.h 库配置文件 控制工程中要启用哪些外设头文件
stm32f10x_it.c 中断服务函数文件 写各种中断处理函数
stm32f10x_it.h 中断头文件 声明中断处理函数

一般做法是在项目根目录的 user 文件夹里放这些文件。

其中，stm32f10x_conf.h 很重要。它的作用是统一包含你要使用的外设头文件。例如你要用 GPIO、串口、定时器，就需要在这里包含对应头文件。

它的理解方式可以记成一句话：

stm32f10x_conf.h 决定了“这个工程准备使用哪些标准外设库”。

添加头文件路径

前面已经把 start 文件夹加入了 Include Paths，后面继续加入 library 和 user。

打开 Keil5 上方魔术棒，进入 C/C++，在 Include Paths 中加入下面几个目录：

路径 作用
.\start 启动文件、系统文件、内核文件所在目录
.\library 标准外设库文件所在目录
.\user 用户代码所在目录

加入后，编译器在处理 #include 时才能正确找到对应头文件。

把文件加入工程分组

把文件复制到文件夹里还不够，还需要在 Keil 工程里把它们真正加入到工程分组中，否则不会参与编译。

通常可以在工程中建立如下几个分组：

分组名 建议加入的文件
start startup_stm32f10x_xx.s、stm32f10x.h、system_stm32f10x.c、system_stm32f10x.h、core_cm3.h、core_cm3.c
library 当前实验需要的标准外设库 .c/.h 文件
user main.c、stm32f10x_conf.h、stm32f10x_it.c、stm32f10x_it.h

需要注意：

  • .h 文件主要用于查看和声明，本身不参与链接
  • 真正参与编译的主要是 .c 和 .s 文件
  • 即使某些 .h 文件不手动加入工程，只要路径正确，理论上也能被 #include 找到
  • 但为了工程结构清晰，通常仍然会把它们也加进分组中方便查看

工程建立完成后的最小结构

一个比较典型的 STM32 标准库工程，最少可以整理成下面这种结构：

文件夹/分组 主要内容 作用
start 启动文件、系统文件、内核文件 保证芯片能启动并完成底层初始化
library 标准外设库文件 提供外设驱动
user 用户代码和中断文件 实现具体功能

最容易遗漏的几个点

新建工程后最常见的问题，基本都出在下面几项：

问题 原因
找不到头文件 Include Paths 没加完整
文件明明复制进文件夹了却不编译 没有真正 Add 到 Keil 工程里
外设函数未定义 对应的 .c 文件没有加入工程
某个头文件报错 标准库版本、CMSIS 版本、编译器版本不兼容
中断函数不生效 启动文件不匹配，或中断服务函数名字写错

新建工程后的最小检查清单

每次新建 STM32 工程后，可以按下面顺序检查一遍：

检查项 是否完成
已复制正确的启动文件到 start
已复制 stm32f10x.h、system_stm32f10x.c、system_stm32f10x.h
已复制 core_cm3.h、core_cm3.c
已建立 user 文件夹并创建 main.c
已建立 library 文件夹并加入需要的外设库文件
已在 Include Paths 中加入 start、library、user
已把 .c/.s 文件真正加入 Keil 工程
已确认启动文件与芯片型号匹配
首次编译通过或已切换到 Compiler Version 5
